# Design: Module 7 Decomposition — Customer Portal

**Date:** 2026-04-25  
**Status:** Approved  
**Scope:** Split BRD Module 7 (Nomination Request / Customer Portal) into 4 focused modules, and create corresponding `module/` directory structure.

---

## Context

BRD Module 7 (Nomination Request — Customer Portal) had grown to cover 6 distinct feature areas: registration/login, nomination form, nomination status tracking, EPB & payment proof upload, voyage tracking, and a customer dashboard. This made it too large for a single Replit handoff or implementation plan.

The decomposition aligns each new module 1:1 with the customer-side swimlane processes defined in `document/Swimlane-Analysis-LPS V3.md`.

## Module Scope Boundaries

### M7 — Customer Authentication & Onboarding
Everything required before a customer can use any feature:
- Self-registration with company data: Customer Name, NPWP, PIC Name, Phone Number, Email, Address, Note; Type is always defaulted to "Cargo Owner" by the system — not shown in the form
- Upload 3 required documents during registration: NPWP file, NIP file, Company Profile file (each with optional Description, Issue Date, Expiry Date)
- Customer Code is auto-generated by the system upon Admin approval — not filled by the customer
- Admin reviews documents, then validates and activates or rejects account
- Login and access to customer dashboard after verification
- Customer account data synced to STS Platform automatically on activation

### M8 — Nomination Request Submission
The nomination creation flow — from blank form to submitted:
- Fill nomination form (Vessel Name, ETA, Cargo Type, Cargo Quantity, Charterer, Est. Barge)
- Upload supporting documents (Rencana Kerja, Shipping Instruction, Surat Penunjukan PBM, Nomor PKK)
- Save as Draft or Submit
- Auto-generate Nomination Number on Submit
- Send nomination data to STS Platform via API
- Ends when nomination is submitted and status is "Submitted"

### M9 — Nomination Status & Payment
Everything that happens after submission, until voyage starts:
- Receive status updates from STS (Pending, Approved, Need Revision)
- "Pending" displays as "Menunggu proses di STS Platform"
- On Approved: view schedule (anchor point, ETB) and EPB detail with amount to pay
- On Need Revision: edit nomination data and re-submit to STS
- Upload Proof of Payment (PDF/JPG/PNG, max 5MB)
- Send payment proof to STS via API → status: "Waiting Payment Verification"
- Receive verification result (Confirmed/Rejected) from STS
- On Rejected: notification with reason + re-upload
- On Confirmed: status → "Payment Confirmed" → voyage starts

### M10 — Customer Dashboard & Monitoring
Customer-facing landing dashboard and ongoing monitoring:
- Customer Portal Dashboard: nomination summary with statuses, active voyages, weather widget
- Nomination Status page: full list of all nominations including Drafts
- Voyage tracking: active voyage with AIS vessel position (view-only, data from STS + LPS AIS)
- Weather & Alert: view real-time weather and alert history for STS Bunati area (view-only)

## Module Dependencies

```
M7 (Auth) ← M8 (Submission) ← M9 (Status & Payment)
                                        ↑
                              M10 (Dashboard) ← consumes M8 + M9 + LPS AIS + Weather API
```

M7 is the entry point with no upstream dependencies. M8 requires M7 (customer must be authenticated). M9 requires M8 (a nomination must exist). M10 is a read-only aggregator that depends on M7 for auth, and consumes data from M8, M9, LPS AIS, and Weather API.
